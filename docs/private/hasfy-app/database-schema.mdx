# Schéma de base de données

## Vue d'ensemble

Hasfy utilise **Supabase** (PostgreSQL) comme base de données principale. L'architecture repose sur Row Level Security (RLS) pour isoler les données par organisation.

## Tables principales

### 1. Authentification et utilisateurs

#### `profiles`

Extension de `auth.users` de Supabase. Contient les informations utilisateur métier.

| Colonne | Type | Description |
|---------|------|-------------|
| `id` | uuid (PK, FK → auth.users) | ID utilisateur Supabase |
| `email` | text | Email de l'utilisateur |
| `full_name` | text | Nom complet |
| `avatar_url` | text | URL de l'avatar (Supabase Storage) |
| `role` | text | Rôle dans l'organisation |
| `created_at` | timestamptz | Date de création |
| `updated_at` | timestamptz | Dernière mise à jour |

#### `roles`

Définition des rôles disponibles dans le système.

| Rôle | Description | Permissions |
|------|-------------|-------------|
| `owner` | Propriétaire de l'organisation | Accès complet, gestion billing |
| `admin` | Administrateur | CRUD complet, gestion utilisateurs |
| `technician` | Technicien | Gestion tickets et équipements |
| `employee` | Employé/Client | Création tickets, vue équipements |

#### `organization_users`

Table de liaison entre utilisateurs et organisations avec le rôle.

| Colonne | Type | Description |
|---------|------|-------------|
| `id` | uuid (PK) | ID unique |
| `user_id` | uuid (FK → profiles) | Utilisateur |
| `organization_id` | uuid (FK → organizations) | Organisation |
| `role` | text | Rôle dans cette organisation |
| `invited_by` | uuid | Qui a invité |
| `invited_at` | timestamptz | Date d'invitation |
| `joined_at` | timestamptz | Date d'acceptation |

## Row Level Security (RLS)

Toutes les tables ont RLS activé. Les politiques isolent les données par organisation :

```sql
-- Exemple de politique RLS sur equipments
CREATE POLICY "Users can view own org equipments"
  ON equipments FOR SELECT
  USING (
    organization_id IN (
      SELECT organization_id FROM organization_users
      WHERE user_id = auth.uid()
    )
  );

CREATE POLICY "Admins can insert equipments"
  ON equipments FOR INSERT
  WITH CHECK (
    organization_id IN (
      SELECT organization_id FROM organization_users
      WHERE user_id = auth.uid()
      AND role IN ('owner', 'admin', 'technician')
    )
  );
```

## Données dans Elasticsearch

Les métriques agents ne sont **pas** stockées dans PostgreSQL mais dans Elasticsearch :

- **Métriques système** : CPU, RAM, disque, réseau
- **Processus** : Liste des processus actifs
- **Sécurité** : Alertes et événements
- **Logs** : Logs système collectés
- **Santé** : État général de l'équipement

L'accès se fait via les indices Elasticsearch, filtré par `organization_id` dans les tags de l'agent.
