## 🎯 Recommandations d'Optimisation

### 🔴 **PRIORITÉ 1: Réduire les Ressources Elasticsearch** (Économie: ~150€/mois)

**Problème actuel:**
```yaml
# Configuration actuelle (SURDIMENSIONNÉE)
Elasticsearch (3 nodes):
  CPU request: 990m × 3 = 2970m (3 vCPU)
  RAM request: 4080Mi × 3 = 12240Mi (~12GB)
  Storage: 10Gi × 3 = 30GB  # ⚠️ Stockage incohérent (30GB vs 300GB PVCs)
```

**Utilisation réelle:**
- CPU utilisé: ~550m (18% des requests)
- RAM utilisée: ~9.5GB (79% des requests)
- **5 nodes GKE** pour supporter ces requests alors que 2-3 suffiraient

**Solution recommandée:**
```yaml
# Configuration optimisée (Phase 1 améliorée)
Elasticsearch (3 nodes):
  CPU request: 500m × 3 = 1500m (1.5 vCPU)
  RAM request: 2Gi × 3 = 6Gi (6GB)
  Storage: 100Gi × 3 = 300GB  # Cohérent avec vos PVCs actuels
  
Économie GKE:
  - RAM économisée: 6GB → réduit de 2-3 nodes
  - CPU économisé: 1.5 vCPU
  - Économie: ~150-200€/mois
```

#### **Action recommandée:**
```bash
# 1. Éditer la configuration Elasticsearch
kubectl edit elasticsearch elasticsearch -n elastic-system

# 2. Modifier les ressources:
spec:
  nodeSets:
  - name: default
    count: 3
    podTemplate:
      spec:
        containers:
        - name: elasticsearch
          resources:
            requests:
              cpu: 500m         # Au lieu de 990m
              memory: 2Gi       # Au lieu de 4080Mi
            limits:
              cpu: 1000m
              memory: 2560Mi
          env:
          - name: ES_JAVA_OPTS
            value: "-Xms1g -Xmx1g"  # Heap = 50% de la RAM
    volumeClaimTemplates:
    - spec:
        resources:
          requests:
            storage: 100Gi    # Au lieu de 10Gi (cohérent avec PVCs)

# 3. Attendre le rolling restart automatique (ECK gère)
# 4. Vérifier la santé: kubectl get es -n elastic-system
```

**Risques:** Aucun - vos ressources actuelles sont largement surdimensionnées.
