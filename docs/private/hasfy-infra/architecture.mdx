# Architecture - Elasticsearch sur GCP

## 🎯 Vue d'ensemble

Infrastructure Elasticsearch progressive sur Google Cloud Platform (GKE Autopilot).

**Architecture évolutive en 4 phases** :
- **Dev** : Environnement développement minimal (210€/mois)
- **Phase 1** : Démarrage production (381€/mois) - 100 agents
- **Phase 2** : Croissance (580€/mois) - 200-500 agents  
- **Phase 3** : Production HA (870€/mois) - 500-1000+ agents

### Architecture Réseau : LoadBalancers

**Avantages vs Ingress** :
- ✅ Compatible certificats auto-signés (L4 passthrough)
- ✅ Configuration simple (pas de BackendConfig/ManagedCertificates)
- ✅ Coût réduit : **21€/mois** (vs 112€ avec Ingress)
- ✅ Port 443 standard (pas de spécification de port dans URLs)
- ✅ Économie : **81% sur infrastructure réseau** (~1092€/an)

**Configuration** :
- 3 LoadBalancers GCP (Network Load Balancer)
- Port 443 standard HTTPS
- Self-signed certificates (ECK managed)
- Pas d'IPs statiques réservées

➡️ Voir [INFRASTRUCTURE.md](INFRASTRUCTURE.md) pour détails complets

### Configuration Elasticsearch

```yaml
apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: elasticsearch
  namespace: elastic-system
spec:
  version: 8.11.0
  nodeSets:
  - name: default
    count: 3
    config:
      node.roles: ["master", "data", "ingest"]
      node.store.allow_mmap: false
    podTemplate:
      spec:
        containers:
        - name: elasticsearch
          resources:
            requests:
              cpu: "2000m"
              memory: "8Gi"
            limits:
              cpu: "2000m"
              memory: "8Gi"
          env:
          - name: ES_JAVA_OPTS
            value: "-Xms4g -Xmx4g"
    volumeClaimTemplates:
    - metadata:
        name: elasticsearch-data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 100Gi
        storageClassName: premium-rwo
    podDisruptionBudget:
      spec:
        minAvailable: 2
```

### Configuration HPA (Horizontal Pod Autoscaler)

```yaml
# HPA pour Data Nodes
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: elasticsearch-data-hpa
  namespace: elastic-system
spec:
  scaleTargetRef:
    apiVersion: elasticsearch.k8s.elastic.co/v1
    kind: Elasticsearch
    name: elasticsearch
  minReplicas: 3
  maxReplicas: 6
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 300
      policies:
      - type: Pods
        value: 1
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 600
      policies:
      - type: Pods
        value: 1
        periodSeconds: 300
```

## 🔄 Migration entre Phases

### Phase 1 → Phase 2

**Durée** : 2-3 heures | **Downtime** : Zéro

Voir [MIGRATION-GUIDE.md](./MIGRATION-GUIDE.md) pour procédure complète.

**Résumé** :
1. Snapshot de sécurité
2. Déployer nouveaux nœuds spécialisés (3 master + 2 data + 1 ingest)
3. Migrer shards progressivement
4. Exclure et supprimer anciens nœuds multi-rôle

### Phase 2 → Phase 3

**Durée** : 1-2 heures | **Downtime** : Zéro

**Résumé** :
1. Augmenter ressources des nœuds existants
2. Activer HPA (Horizontal Pod Autoscaler)
3. Configurer multi-zone si pas déjà fait
4. Mettre à jour load balancers

## 🌍 Multi-région (Phase 3+)

Cross-Cluster Replication possible pour DR :

```
Primary Region (europe-west9)
        ↓ CCR
Secondary Region (europe-west1)
```

Configuration CCR :

```bash
# Sur cluster secondaire
PUT /_ccr/follow/my-index
{
  "remote_cluster": "primary",
  "leader_index": "my-index",
  "settings": {
    "index.number_of_replicas": 1
  }
}
```
