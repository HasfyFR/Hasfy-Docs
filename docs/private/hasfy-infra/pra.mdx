# Plan de Reprise d'Activité (PRA) - Elasticsearch sur GCP

## 📋 Définitions

**PRA (Plan de Reprise d'Activité)** : Stratégie organisationnelle pour maintenir ou restaurer les activités métier critiques après une interruption majeure.

**Différence PRI vs PRA** :
- **PRI** : Procédures techniques (comment restaurer les systèmes IT)
- **PRA** : Procédures métier (comment continuer l'activité business)

**Objectifs** :
- **MTPD (Maximum Tolerable Period of Disruption)** : 8 heures
- **RTO (Recovery Time Objective)** : 4 heures
- **RPO (Recovery Point Objective)** : 1 heure
- **MBCO (Minimum Business Continuity Objective)** : 80% capacité

## 🚨 Classification des incidents

### Niveau 1 : Mineur
**Caractéristiques** :
- Dégradation performance (&lt;20%)
- Pas d'impact utilisateur visible
- Services essentiels fonctionnels

**Exemples** :
- 1 pod Elasticsearch en erreur (cluster reste green)
- Latence +50ms
- Warning logs

**Actions** :
- Monitoring renforcé
- Investigation en horaires bureau
- Pas d'escalade

---

### Niveau 2 : Modéré
**Caractéristiques** :
- Dégradation significative (20-50%)
- Impact utilisateur limité
- Services dégradés mais opérationnels

**Exemples** :
- Cluster Elasticsearch yellow
- 50% des requêtes lentes
- Kibana intermittent

**Actions** :
- Notification équipe technique (30 min)
- Investigation prioritaire
- Communication interne
- Escalade L2 si pas résolu en 2h

**RTO** : 4 heures  
**RPO** : 1 heure

---

### Niveau 3 : Majeur
**Caractéristiques** :
- Service partiellement indisponible (>50%)
- Impact métier significatif
- Perte de données possible

**Exemples** :
- Cluster Elasticsearch red
- Kibana inaccessible
- Ingestion logs en échec
- Perte d'un datacenter GCP

**Actions** :
- Activation cellule de crise immédiate
- Notification parties prenantes (15 min)
- Communication externe clients (1h)
- Activation procédures PRI
- Escalade L3 + support éditeurs

**RTO** : 2 heures  
**RPO** : 30 minutes

---

### Niveau 4 : Critique (Désastre)
**Caractéristiques** :
- Service totalement indisponible
- Impact métier critique
- Perte région complète
- Compromission sécurité

**Exemples** :
- Destruction infrastructure GCP région
- Cyberattaque ransomware
- Corruption données généralisée
- Panne électrique datacenter prolongée

**Actions** :
- **Activation PRA immédiate**
- Cellule de crise H24
- Communication officielle (CEO)
- Activation site de secours
- Support éditeurs mode urgence
- Possibilité recours procédures légales

**RTO** : 8 heures (MTPD)  
**RPO** : 24 heures (backup cross-region)

## 🔄 Stratégies de continuité

### Stratégie 1 : Mode dégradé (Incident Niveau 2)

**Scénario** : Cluster yellow, performance réduite

**Actions métier** :

1. **Priorisation des flux** :
   ```bash
   # Désactiver ingestion non-critique
   curl -k -X PUT -u "elastic:$PASSWORD" \
     https://elastic.hasfy.fr/_cluster/settings \
     -H "Content-Type: application/json" \
     -d '{
       "transient": {
         "cluster.routing.allocation.enable": "primaries"
       }
     }'
   
   # Limiter les dashboards Kibana gourmands
   # Désactiver les requêtes non-urgentes
   ```

2. **Communication utilisateurs** :
   ```
   Objet : Dégradation service Elasticsearch
   
   Chers utilisateurs,
   
   Nous rencontrons actuellement une dégradation de performance 
   sur notre plateforme Elasticsearch (~30% plus lent).
   
   Services disponibles :
   - ✅ Ingestion logs (prioritaire)
   - ✅ Recherche logs (ralenti)
   - ⚠️ Dashboards complexes (peut échouer)
   
   Actions en cours : Investigation + scaling
   ETA résolution : 2 heures
   
   Prochaine mise à jour : dans 1h
   ```

3. **Alternatives temporaires** :
   - Logs applicatifs locaux (fallback)
   - Export CSV pour analyses urgentes
   - Accès direct logs GCP Cloud Logging

**Seuils de retour à la normale** :
- Cluster status = green
- Latence P95 &lt; 500ms
- Tous les shards assigned

### Stratégie 2 : Basculement région (Incident Niveau 3-4)

**Scénario** : Région GCP europe-west9 (Paris) indisponible

#### Phase 1 : Activation site de secours (1 heure)

**Prérequis** (configuration préventive) :

```bash
# 1. Infrastructure Terraform région DR
cd providers/gcp/terraform-dr

cat > terraform.tfvars <<EOF
project_id = "YOUR_PROJECT_ID"
region = "europe-west1"  # Belgique
zone = "europe-west1-b"
cluster_name = "elastic-dr"
environment = "disaster-recovery"
EOF

# Déployer infrastructure DR (à faire AVANT incident)
terraform apply

# 2. Configuration réplication DNS
# Zone DNS avec failover automatique
gcloud dns record-sets create kibana.hasfy.fr. \
  --zone=hasfy-zone \
  --type=A \
  --ttl=300 \
  --routing-policy-type=GEO \
  --routing-policy-data="europe-west9=IP_PRIMARY;europe-west1=IP_DR" \
  --enable-health-checking
```

**Lors de l'incident** :

```bash
# 1. Vérifier disponibilité infrastructure DR
gcloud container clusters describe elastic-dr --region=europe-west1

# 2. Restaurer données depuis backup cross-region
gsutil ls gs://elastic-backups-dr/snapshots/

# 3. Déployer stack Elasticsearch sur cluster DR
kubectl config use-context gke_YOUR_PROJECT_europe-west1_elastic-dr

cd kubernetes
kubectl apply -f 01-eck-operator/
kubectl apply -f 02-elasticsearch/
# ... suite déploiement (voir PRI.md Scénario 2)

# 4. Restaurer dernier snapshot
LAST_SNAPSHOT=$(gsutil cat gs://elastic-backups-dr/snapshots/latest.txt)
curl -k -X POST -u "elastic:$PASSWORD" \
  https://elastic-dr.hasfy.fr/_snapshot/gcs-repo-dr/$LAST_SNAPSHOT/_restore

# 5. Basculer DNS
gcloud dns record-sets update kibana.hasfy.fr. \
  --zone=hasfy-zone \
  --type=A \
  --rrdatas=IP_DR \
  --ttl=60
```

### Stratégie 3 : Mode survie (Incident Niveau 4 prolongé)

**Scénario** : Impossibilité de restaurer service Elasticsearch rapidement

**Solutions de contournement** :

#### Option A : Logs locaux applications

```bash
# Configurer applications pour logger localement
# Example : application Node.js

# .env
ELASTICSEARCH_ENABLED=false
LOG_TO_FILE=true
LOG_FILE_PATH=/var/log/app/app.log

# Rotation logs
logrotate -f /etc/logrotate.d/app
```

#### Option B : GCP Cloud Logging (temporaire)

```bash
# Rediriger logs vers Cloud Logging
gcloud logging write app-logs "Emergency log entry" \
  --severity=ERROR \
  --resource=global

# Consulter
gcloud logging read "resource.type=global AND severity>=ERROR" \
  --limit=50 \
  --format=json

# Export vers BigQuery pour analyse
bq mk --dataset logs_emergency
gcloud logging sinks create emergency-logs \
  bigquery.googleapis.com/projects/YOUR_PROJECT/datasets/logs_emergency
```

#### Option C : Service Elasticsearch tiers (ultime recours)

**Elastic Cloud** :

```bash
# 1. Créer déploiement Elastic Cloud (15 min)
# Via https://cloud.elastic.co

# 2. Restaurer depuis backup GCS
# Télécharger snapshots
gsutil -m cp -r gs://elastic-backups-primary/snapshots/ /tmp/snapshots/

# 3. Uploader vers Elastic Cloud
# Utiliser elastic-cloud-cli ou UI
```

**Coût** : ~800 EUR/mois (vs 562 EUR self-hosted)  
**Durée recommandée** : Maximum 1 mois

### Test mensuel léger (1 heure)

**Objectif** : Vérifier snapshots et procédures basiques

```bash
#!/bin/bash
# test-pra-light.sh

echo "=== Test PRA mensuel $(date) ==="

# 1. Vérifier derniers snapshots
echo "[1/5] Vérification snapshots..."
LAST_SNAPSHOT=$(curl -sk -u "elastic:$PASSWORD" \
  https://elastic.hasfy.fr/_snapshot/gcs-repo/_all | \
  jq -r '.snapshots | sort_by(.start_time_in_millis) | last | .snapshot')

echo "Dernier snapshot : $LAST_SNAPSHOT"

# 2. Vérifier backup cross-region
echo "[2/5] Vérification backup DR..."
gsutil ls gs://elastic-backups-dr/snapshots/ | tail -5

# 3. Tester restauration index test
echo "[3/5] Test restauration..."
curl -sk -X POST -u "elastic:$PASSWORD" \
  "https://elastic.hasfy.fr/_snapshot/gcs-repo/$LAST_SNAPSHOT/_restore" \
  -H "Content-Type: application/json" \
  -d '{"indices": "test-*", "rename_pattern": "(.+)", "rename_replacement": "restored_$1"}'

# 4. Vérifier infrastructure DR
echo "[4/5] Vérification infra DR..."
gcloud container clusters describe elastic-dr --region=europe-west1 --format="value(status)"

# 5. Valider contacts cellule de crise
echo "[5/5] Validation contacts..."
# Envoyer email test
echo "Test PRA automatique" | mail -s "Test PRA $(date)" cto@hasfy.fr
```
