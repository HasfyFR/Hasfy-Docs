# Infrastructure LoadBalancer - Hasfy Elastic Stack

## 📍 Architecture Actuelle (Production)

L'infrastructure Elasticsearch est déployée sur **GKE Autopilot** avec une architecture **LoadBalancer** pure, optimisée pour les coûts et la simplicité.

### ✅ Services exposés

| Service | DNS | IP externe | Port | Type |
|---------|-----|------------|------|------|
| **Elasticsearch** | `elastic.hasfy.fr` | `34.155.235.161` | 443 | HTTPS (LoadBalancer) |
| **Fleet Server** | `fleet.hasfy.fr` | `34.155.81.172` | 443 | HTTPS (LoadBalancer) |
| **Kibana** | `kibana.hasfy.fr` | `34.163.3.153` | 443 | HTTPS (LoadBalancer) |

**Caractéristiques** :
- ✅ Port 443 standard (HTTPS sans spécification de port)
- ✅ Certificats auto-signés (ECK managed)
- ✅ LoadBalancer L4 (passthrough, pas de validation SSL côté GCP)
- ✅ Architecture simplifiée et économique

```bash
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}'

  -o jsonpath='{.status.loadBalancer.ingress[0].ip}'

  -o jsonpath='{.status.loadBalancer.ingress[0].ip}'
```

```bash

kubectl get secret elasticsearch-es-elastic-user \
  -n elastic-system \
  -o go-template='{{.data.elastic | base64decode}}'
```

## 🔐 Sécurité

### Certificats SSL

Tous les services utilisent des **certificats auto-signés** générés automatiquement par l'opérateur ECK :

- **Elasticsearch** : `elasticsearch-es-http-certs-public`
- **Fleet Server** : `fleet-server-agent-http-certs-public`
- **Kibana** : `kibana-kb-http-certs-public`

### Authentification

| Service | Méthode | Détails |
|---------|---------|---------|
| **Elasticsearch** | Basic Auth | User: `elastic`, Password: dans Secret |
| **Fleet Server** | Token-based | Token d'enrollment pour agents |
| **Kibana** | Session-based | Via Elasticsearch authentication |

### Configuration `-k` (Insecure)

Les commandes `curl` utilisent l'option `-k` car les certificats sont auto-signés. En production avec certificats valides (Let's Encrypt), cette option ne sera plus nécessaire.

## 💰 Analyse des Coûts

### Architecture Précédente (Ingress)

| Ressource | Quantité | Coût unitaire | Total |
|-----------|----------|---------------|-------|
| Ingress (HTTP(S) LB) | 4 | ~20€/mois | ~80€ |
| IPs statiques globales | 4 | ~8€/mois | ~32€ |
| **Total Ingress** | | | **~112€/mois** |

**Problèmes** :
- ❌ Backends UNHEALTHY (incompatibilité certificats auto-signés)
- ❌ Configuration complexe (BackendConfig, ManagedCertificates)
- ❌ Coût élevé (4 Ingress + 4 IPs statiques)

### Architecture Actuelle (LoadBalancer)

| Ressource | Quantité | Coût unitaire | Total |
|-----------|----------|---------------|-------|
| LoadBalancer (Network LB) | 3 | ~7€/mois | ~21€ |
| **Total LoadBalancer** | | | **~21€/mois** |

**Avantages** :
- ✅ Compatible certificats auto-signés (L4 passthrough)
- ✅ Configuration simple (une ligne Kubernetes)
- ✅ Coût réduit (pas d'IPs statiques réservées)

### 📊 Économie Réalisée

- **Économie mensuelle** : ~91€/mois (81% de réduction)
- **Économie annuelle** : ~1092€/an
- **ROI** : Immédiat (migration sans coût)

## 🔄 Migration Ingress → LoadBalancer

### Pourquoi Migrer ?

1. **Backends UNHEALTHY** : Les Ingress GCP ne supportent pas bien les certificats auto-signés
2. **Coût élevé** : 4 Ingress (80€) + 4 IPs statiques (32€) = 112€/mois
3. **Complexité** : BackendConfig, ManagedCertificates, health checks complexes

### Processus de Migration

```bash
# 1. Créer LoadBalancers
kubectl patch svc elasticsearch-es-http -n elastic-system \
  -p '{"spec": {"type": "LoadBalancer"}}'

kubectl patch svc fleet-server-agent-http -n elastic-system \
  -p '{"spec": {"type": "LoadBalancer"}}'

kubectl patch svc kibana-kb-http -n elastic-system \
  -p '{"spec": {"type": "LoadBalancer"}}'

# 2. Configurer port 443 (voir fichiers YAML dans kubernetes/)

# 3. Attendre assignation IPs externes (2-3 minutes)
kubectl get svc -n elastic-system -w

# 4. Tester connectivité avec nouvelles IPs

# 5. Supprimer Ingress et ressources associées
kubectl delete ingress -n elastic-system --all
kubectl delete managedcertificate -n elastic-system --all
kubectl delete backendconfig -n elastic-system --all
gcloud compute addresses delete [IP_NAMES] --global
```
